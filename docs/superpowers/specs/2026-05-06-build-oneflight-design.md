# Build Oneflight Design

- Date: 2026-05-06
- Issue: https://github.com/windoliver/agentenv/issues/65
- Milestone: M2 drivers / hardening
- Affected crates: `agentenv-core`, `sandbox-openshell`, `agentenv-events`, `agentenv`
- Protocol impact: no driver protocol or schema-version change

## 1. Context And Goals

Issue #65 asks agentenv to avoid duplicate sandbox image builds when many concurrent create requests target equivalent environments. The repository currently has one concrete build path: OpenShell sandbox creation with `sandbox.image.source: byo`. In that path, `agentenv-core` passes BYO Dockerfile metadata in `SandboxSpec.metadata`, and `sandbox-openshell` stages the Dockerfile context, runs `docker build`, records `image-digest`, then calls `openshell sandbox create`.

The goal is to add digest-keyed oneflight around that build step, not around sandbox creation. Each `create` still gets its own sandbox handle. Only the reusable BYO image materialization is shared.

## 2. Scope And Non-Goals

In scope:

1. Compute a deterministic build key for OpenShell BYO image materialization.
2. Ensure concurrent creates for the same build key perform one Docker build.
3. Let waiters reuse the built staged context and computed digest.
4. Persist build metadata under the runtime root for future reuse.
5. Safely invalidate stale cache entries and retry one fresh build.
6. Add bounded queue controls for concurrent builds.
7. Expose the required Prometheus metrics:
   - `agentenv_build_oneflight_hits_total`
   - `agentenv_build_oneflight_misses_total`
   - `agentenv_build_queue_depth`

Out of scope:

1. Fleet-wide scheduling across hosts.
2. Full sandbox pooling.
3. Adding protocol methods such as `materialize_build`.
4. Making non-OpenShell or future sandbox drivers implement build caching before they have a real build path.
5. Sharing sandbox handles between create requests.

## 3. Recommended Architecture

Use a two-layer key and keep the oneflight implementation in `sandbox-openshell`.

`agentenv-core` adds a stable build-key seed to sandbox metadata when the sandbox image is BYO. The seed is a `sha256:` digest over data core already owns:

1. the verified blueprint YAML after lifecycle validation,
2. the lockfile YAML that create will persist,
3. resolved sandbox, agent, context, and inference driver identities,
4. relevant sandbox metadata generated by core, including agent name, MCP port, workspace mount, and agentenv version.

Core writes these metadata keys:

1. `agentenv_build_oneflight`: `"byo-openshell-v1"`,
2. `agentenv_build_seed`: the core seed digest,
3. `agentenv_build_seed_version`: `"1"`.

Non-BYO creates omit all three keys.

`sandbox-openshell` completes the final build key by hashing the seed plus the actual BYO build inputs it owns:

1. canonical Dockerfile path,
2. Dockerfile bytes,
3. staged context file paths, file bytes, file modes where available, and symlink targets,
4. build args passed to Docker,
5. OpenShell driver version.

This split avoids a protocol change and keeps context-staging semantics in the driver, where `.dockerignore` and filesystem behavior already live.

## 4. Build Cache Layout

Persist build metadata under the runtime root:

```text
<root>/build-cache/
  <build-key>/
    metadata.json
    context/
      Dockerfile
      ...
    image-digest
    lock/
```

`metadata.json` contains:

```json
{
  "version": 1,
  "build_key": "sha256:...",
  "driver": "openshell",
  "driver_version": "0.0.1-alpha0",
  "image_ref": "/Users/alice/.agentenv/build-cache/sha256-.../context",
  "image_digest": "sha256:...",
  "created_at": "2026-05-06T00:00:00Z",
  "source": {
    "dockerfile": "/abs/path/Dockerfile",
    "context_digest": "sha256:..."
  }
}
```

The `image_ref` remains the staged context path because OpenShell create currently receives the staged directory through `--from`. The Docker tag should become key-based, such as `agentenv-byo-<short-key>:latest`, rather than environment-name-based.

For compatibility with existing core lockfile update logic, each create also writes or copies the shared `image-digest` to the current per-env sidecar path:

```text
<root>/build/<env-name>/image-digest
```

This lets `agentenv-core::record_computed_byo_image_digest` keep recording the computed digest into the created environment lockfile.

## 5. Oneflight Semantics

`sandbox-openshell` owns a `BuildCoordinator` with file-backed locking so separate driver instances in the same process and separate CLI processes coordinate through the same runtime root.

For a BYO create:

1. Compute the final build key.
2. Validate an existing cache entry.
3. If valid, count a oneflight hit, copy the digest to the per-env sidecar, and use cached `image_ref`.
4. If missing or invalid, try to become the builder by atomically creating the key lock directory.
5. The builder stages context, runs `docker build`, inspects the digest, writes metadata atomically, removes the lock, and proceeds with sandbox creation.
6. Waiters that find the lock wait for metadata or failure with a bounded polling interval.
7. If the builder fails, it writes a small failure marker with a sanitized reason and releases the lock. Waiters surface the same reason.
8. If a stale lock exceeds the configured timeout, one waiter may take ownership after validating that no usable metadata exists.

The oneflight result is the materialized image reference and digest. The sandbox create call still runs once per create request.

## 6. Bounded Queue Controls

Add OpenShell driver controls, read from environment variables during driver initialization:

1. `AGENTENV_BUILD_MAX_INFLIGHT`: maximum distinct build keys building at once. Default: `4`.
2. `AGENTENV_BUILD_QUEUE_LIMIT`: maximum waiters across all build keys. Default: `128`.
3. `AGENTENV_BUILD_LOCK_TIMEOUT_SECS`: stale lock timeout. Default: `900`.

When the queue limit is exceeded, create fails before sandbox creation with `DriverError::PreflightFailed` or `DriverError::InvalidInput` carrying a clear queue-saturated message. No credentials are included in queue errors.

The first implementation can enforce the per-process queue limit with an async semaphore and the per-key oneflight with filesystem locks. The filesystem lock still protects cross-process duplicate builds. If a second process exceeds its own local queue limit, it fails locally rather than corrupting the shared cache.

## 7. Cache Validation And Invalidation

A cache entry is valid only when:

1. `metadata.json` parses and has `version: 1`.
2. `build_key` matches the expected key.
3. `driver` is `openshell`.
4. `driver_version` matches the running driver version.
5. `image_ref` exists and points inside `<root>/build-cache/<key>/context`.
6. `image-digest` exists, matches metadata, and parses as `sha256:`.
7. `docker image inspect agentenv-byo-<short-key>:latest --format {{.Id}}` succeeds and returns the same digest.

If validation fails, the driver removes the invalid metadata file and tries one fresh build. If the fresh build also fails, the create fails with that build error. Waiters receive the same build error.

## 8. Metrics And Events

Extend `agentenv-events` with internal activity kinds for build oneflight observation:

1. `BuildOneflightHit`
2. `BuildOneflightMiss`
3. `BuildQueueDepth`

`sandbox-openshell` receives the production event emitter from `BuiltInDriverFactory`, just as subprocess agent and context drivers already do. It emits:

1. a hit event when a request reuses valid metadata or waits behind an active builder,
2. a miss event when a request becomes the builder,
3. queue-depth events when waiter count changes, with `extras.depth` set to the current global waiter count for the process.

`agentenv-events::metrics::MetricsSnapshot` derives:

1. `agentenv_build_oneflight_hits_total` from hit events,
2. `agentenv_build_oneflight_misses_total` from miss events,
3. `agentenv_build_queue_depth` from the latest queue-depth event, defaulting to zero.

The metrics are global and should not include build-key labels. Build keys are digests of local filesystem inputs and can be high-cardinality.

## 9. Error Handling

Builder failures propagate to all waiters as the same sanitized `DriverError`. The stored failure marker includes only:

1. build key,
2. timestamp,
3. stable reason code,
4. redacted message.

Raw Docker stdout and stderr remain in the immediate driver error path but must not be persisted in cache metadata. If a create fails after the build succeeds but before sandbox creation completes, the build metadata remains reusable because sandbox creation is separate from image materialization.

Rollback remains owned by `agentenv-core`: if sandbox creation fails after a successful build, core tears down provisioned context and inference resources as it does today.

## 10. Testing Strategy

Use test-driven implementation.

`sandbox-openshell` tests:

1. concurrent creates with the same BYO inputs run one `docker build` and multiple `openshell sandbox create` commands,
2. a cached valid metadata entry skips `docker build`,
3. an invalid digest or missing Docker image evicts metadata and rebuilds once,
4. a builder failure wakes waiters with the same clear reason,
5. queue limit rejection happens before `openshell sandbox create`,
6. per-env `build/<name>/image-digest` sidecars are written for both builder and cache-hit creates.

`agentenv-core` tests:

1. BYO sandbox specs include the deterministic build-key seed metadata,
2. non-BYO sandbox specs do not include build oneflight metadata,
3. the existing computed digest lockfile behavior still records `sandbox-image`.

`agentenv-events` tests:

1. activity kinds serialize to stable snake_case labels,
2. Prometheus rendering includes the three required metrics,
3. queue depth uses the latest queue-depth event and defaults to zero.

Full workspace verification remains:

```bash
cargo fmt --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
```

## 11. Trade-Offs

This design deliberately avoids a driver protocol bump. That means the OpenShell driver owns the concrete oneflight behavior, and future sandbox drivers will need their own implementation or a later protocol-level materialization API. That trade-off is acceptable for issue #65 because OpenShell BYO images are the only actual build path in the current tree.

The cache is local to one runtime root. It does not coordinate across machines, remote hosts, or fleet schedulers. That matches agentenv's current env-manager scope and the issue non-goals.

The queue-depth metric is event-derived because `agentenv metrics serve` reads the activity store from a separate process. It reflects the latest observed local queue depth rather than querying a daemon.
